The Presence Engine

A multi-layered intimate interaction engine that translates physical touch, sound, and device motion into a synchronized sensory experience across any distance. Physics simulation, low-latency audio pipeline, and haptic synthesis — all operating simultaneously.

2D Soft-Body Physics Mesh

The interaction surface is a live physical model — vertices connected by elastic springs governed by Hooke's Law. Touch input deforms the mesh in real time using Android's multi-touch telemetry: contact ellipse footprint, pressure magnitude, and contact angle.

  • Hooke's Law anchor springs for elastic restoration
  • Multi-touch repulsion from finger contact ellipses
  • Accelerometer gravity vector shifts mesh sag in real time

ASMR Whisper Mode

A parallel low-latency WebRTC voice link configured for intimate, close-range audio. Standard mobile processing — echo cancellation, noise suppression, automatic gain control — is bypassed entirely to stream raw, uncompressed audio. Playout via device earpiece by default.

  • Raw WebRTC audio — no AGC, no noise suppression
  • Earpiece routing for private listening
  • Simultaneous to physics simulation, zero blocking

Audio-Tactile Synthesis & Mutual Press

The physics mesh continuously calculates Kinetic Energy and Mesh Stress. A built-in synthesizer maps kinetic energy → volume and mesh stress → sub-bass pitch. When both partners touch simultaneously, the system enters Mutual Press — a haptic loop whose frequency blends both partners' pressure into a personalized tactile pulse.

  • Kinetic energy → volume; mesh stress → pitch
  • Mutual Press detection via WebSocket telemetry
  • Personalized haptic heartbeat blended from both inputs

Discreet Privacy Signals

When the canvas is opened, a PRESENCE_INVITATION packet is dispatched via WebSocket relay. If your partner is inactive, a high-priority system notification fires — disguised with a generic compass icon and the text "Someone is thinking of you… ❤️", revealing neither sender identity nor app name.

  • Steganographic notification design
  • No sender identity in push payload
  • Ntfy self-hosted silent wakeup — no third-party push

Setup Guide — Practical Path

A comprehensive walkthrough to provision, configure, and compile your sovereign communication stack.

⚙️ Prerequisites & Project Scope

What is required to host your secure Enclave, and what you can expect from the setup workflow.

System Requirements

  • Linux VPS: Ubuntu/Debian server with minimum 1 vCPU & 2GB RAM.
  • DNS Domain: A custom domain name pointing records to your server.
  • Android Environment: Android Studio or SDK CLI (to compile client app).

What You Can Expect

  • Time Investment: ~15 to 20 minutes for full server & mobile setup.
  • Absolute Control: 100% data residency — keys, databases, and assets stay strictly on your hardware.
  • High Reliability: WebRTC peer connection traverse even strict carrier NATs using Coturn.
saif@enclave-vps: ~

Step 1: Clone the repository and scaffold template configurations

Clone the core workspace repository onto your local environment, enter the folder, and bootstrap configuration files.

# Clone Enclave server and mobile codebase
git clone https://github.com/saifmukhtar/enclave.git
cd enclave

# Scaffold properties and local environment template templates
cp enclave-ui/local.properties.example enclave-ui/local.properties
cp enclave-server/.env.example enclave-server/.env

Step 2: Bootstrap the local Dockerized backend stack

Give execution rights to our local script and run it. This will automatically spin up PostgreSQL database containers, Supabase Realtime listeners, Coturn TURN server relays, and Ntfy secure local notifications.

# Elevate execution permissions
chmod +x setup-local.sh

# Bootstrap PostgreSQL, Supabase engine, and secure local relays
./setup-local.sh

Step 3: Configure Android client environment variables

Open enclave-ui/local.properties and fill in your SDK path, self-hosted Supabase access credentials, Node.js WebSocket signaling server address, Coturn credentials, and private self-hosted Ntfy push configurations.

# Android SDK paths and dynamic access variables
sdk.dir=/home/username/Android/Sdk
SUPABASE_URL=https://your-supabase-instance.com
SUPABASE_KEY=your-anon-key
SIGNALING_SERVER_URL=wss://wss.yourdomain.com
TURN_SERVER_URL=turn:your-vps-ip:3478
NTFY_SERVER_URL=https://ntfy.yourdomain.com
NTFY_USERNAME=your-ntfy-user
NTFY_PASSWORD=your-ntfy-pass

Step 4: Compile and install the Android app

Run the Gradle assembler directly from project directory to build a signed release bundle or a local debug build.

# Navigate into the mobile Android client subdirectory
cd enclave-ui

# Compile a local debug APK for testing
./gradlew assembleDebug

# Or assemble a production-grade signed release APK
./gradlew assembleRelease

Step 5: Cloud Deployment & VPS Production Hardening

Map DNS, provision LetsEncrypt TLS encryption via Certbot certificates, lock down firewalls, and launch signaling endpoints under PM2.

# Navigate into the self-hosted server stack folder
cd enclave-server

# Execute deployment workflow script to deploy securely to your VPS
./deploy_to_cloud.sh

See the complete SETUP_GUIDE.md on GitHub for Coturn, Nginx server blocks, and Certbot automation.

Security Model

🚫

Client-Side Email Whitelist

Registration on your Supabase instance is restricted to whitelisted email addresses — preventing any third party from joining your private instance.

🔗

Deterministic Peer Pairing

Pairing logic is deterministic and hardcoded client-side: User A always resolves User B, and vice versa. No ambiguity, no discovery surface.

🔒

Permanent Lockdown

Once both partners register, disable signups in Supabase Auth settings entirely. The database becomes permanently sealed against new accounts.

🛡️

Zero-Knowledge Keys

Encryption keys are derived locally and never transmitted. PBKDF2 key derivation from your master passphrase means only you can decrypt your vault and backups.

Open Cryptographic Protocol Specifications

Enclave leverages proven mathematical models to ensure that communication remains perfectly sovereign, zero-knowledge, and isolated from external networks.

Extended Triple Diffie-Hellman (X3DH)

Before A communicates with B, they negotiate a shared master key using the X3DH key exchange protocol. This protocol provides high identity verification security, forward secrecy, and complete cryptographic deniability.

Curve25519SHA-256Identity Keys
  • Uses ephemeral keypairs for mathematical isolation of key derivation paths.
  • Signed PreKeys prevent man-in-the-middle spoofing vectors.
  • One-Time Prekeys allow asynchronous encrypted pairing.

Double Ratchet Key Rotation

Once keys are exchanged, messages are ratcheted using a dual-state KDF Chain Ratchet (Diffie-Hellman + Symmetric ratcheting). This ensures that every single message payload is sealed with a unique, self-destructing key.

HKDF-SHA256AES-256-CBCHMAC-SHA256
  • Forward Secrecy: Passive listeners cannot deduce past messages using present keys.
  • Break-in Secrecy: Key compromise automatically heals inside subsequent roundtrips.
  • Out-of-order message handling is supported via pre-calculated ratchet skips.

Zero-Knowledge Vault Backup & Storage

All sensitive gallery and database assets are sealed behind hardware-accelerated local encryption. Master key derives on-device via PBKDF2, ensuring data in transit is opaque to your self-hosted servers.

AES-256-GCMPBKDF2Keystore HSM
  • Local storage locked down inside an encrypted WAL Room SQLite database.
  • Passphrase is salted and iterated 100,000 times before AES-256-GCM envelope sealing.
  • Supports biometric auth tied directly to Android Keystore Hardware Backed keys.

Sovereign Notification Wakes

Avoids centralized Google FCM gateways. Uses an isolated, continuous secure Ntfy relay to trigger silent system wakeups and sync incoming E2EE keys instantly.

NtfyWebSocket RelayForeground Wakes
  • Silent background sync wakes client logic instantly upon inbound payloads.
  • Absolutely zero tracking metadata or telemetry exposed to third-party providers.
  • Encrypted payload flags ensure that only notifications are shown client-side.
